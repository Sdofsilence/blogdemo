# tcp三次握手 四次挥手

标志字段（Flags）是一个6位字段，包含六个控制位，分别是,这些标志字段共同决定了TCP连接的状态和行为。

* URG（紧急，Urgent）： 当URG=1时，表示紧急指针字段有效，用于指示紧急数据。
* ACK（确认，Acknowledgment）： 当ACK=1时，表示确认号字段有效。
* PSH（推送，Push）： 当PSH=1时，表示接收端应立即将数据交付给上层应用。
* RST（重置，Reset）： 当RST=1时，表示连接需要重置。
* SYN（同步，Synchronize）： 当SYN=1时，表示请求建立连接。
* FIN（结束，Finish）： 当FIN=1时，表示请求断开连接。

##### 为什么要三次握手，前两次是必要的，重点是第三次？为了防止syn flood攻击；

**第三次握手至关重要，它解决了网络中的一个核心风险：过期的重复连接请求**。

想象一下这个场景，如果没有第三次握手，会发生什么：

### 问题根源：延迟的“幽灵连接”

网络是不稳定的，数据包可能会“迷路”，在网络中游荡很久才到达。

1. 客户端之前发送了一个 `SYN` 包（第一次握手）请求建立连接，但这个包被网络节点延迟了，成了一个 **“过期”的包**。
2. 客户端等不到服务器的回应，超时后重发了一个新的 `SYN` 包，这次成功建立了连接并完成了通信，连接已关闭。
3. 此时，那个**迷路的、过期的 `SYN` 包** 终于到达了服务器。

**如果没有第三次握手，服务器会如何反应？**

* 服务器收到这个过期的 `SYN`，认为这是一个新的连接请求，于是回应 `SYN-ACK`（第二次握手），并**单方面为其分配资源，认为连接已经建立**。
* 服务器会一直等待客户端发送数据，但客户端对此一无所知（因为它早就完成了之前的请求），永远不会发送数据。这就导致服务器上维护着一个 **“幽灵连接”**，白白消耗着CPU、内存等资源。如果这种攻击被恶意利用，大量伪造的 `SYN` 包会导致服务器资源耗尽，无法服务正常请求，这就是著名的 **SYN Flood 攻击**。

### 第三次握手如何解决这个问题？

现在，我们引入第三次握手，看看同样的场景会怎样：

1. 过期的 `SYN`（第一次握手）到达服务器。
2. 服务器回应 `SYN-ACK`（第二次握手）。
3. **关键步骤**：服务器在收到客户端的 **`ACK`（第三次握手）之前，并不会完全打开连接**。此时的连接处于一种“半开”状态，资源分配是谨慎且有限的。
4. 客户端收到这个 `SYN-ACK` 后，会检查这个连接是否是自己发起的。它发现这个 `SYN-ACK` 是对一个它根本不记得的、过期的连接的回应。
5. 因此，客户端**不会发送第三次握手的 `ACK` 确认**，而是会发送一个 `RST`（复位）包来告知服务器重置这个连接。
6. 服务器收到 `RST` 后，释放之前分配的资源，避免了资源的浪费。

**只有当一个 `SYN-ACK` 被一个合法的 `ACK` 确认后，服务器才100%确信：对方不仅在线，而且它确实有意向建立这个连接，并且连接请求是当前有效的，而不是一个网络延迟造成的“幽灵”。**

### LACK_ACK

* **RTO计时器**：当被动关闭方发送FIN报文后进入LAST_ACK状态，会启动一个RTO计时器，等待对方的最后一次ACK报文。RTO时间根据RTT（round trip time）动态计算，通常几秒。

* **若收到ACK**：被动关闭方会结束LAST_ACK，进入到CLOSED状态，正常关闭连接。
* **若RTO时间没有收到ACK**：被动关闭方会认为之前发送的FIN报文可能丢失了，于是会重传FIN报文，并重新启动RTO计时器。
* **多次重传后仍未收到ACK**：经过若干次重传（3-5）次后，如果还没有收到ACK，被动关闭方最终会放弃等待，发送RST报文来强制关闭连接，结束LACK_ACK状态。
