## 公钥加密基础设施(PKI) 数字证书颁发机构（CA）或者说tls的原理：

* PKI概念介绍<<https://www.cnblogs.com/lianshuiwuyi/p/17766615.html>>
* 原理：
  * 证书：一个网站需要向一个可信的CA机构申请网站的证书（其中包含自己的 网站-公钥），CA机构会用自己的 CA-私钥 签名这个证书，然后网站把证书放到自己的服务中
  * 验签：TLS连接的时候，网站把证书发给客户端，客户端识别CA机构，并在（根证书：预先安装在系统和浏览器中）中找到CA机构的 CA-公钥对证书中的签名进行解密，获得证书指纹，自己计算一遍证书指纹，两个对比验证证书
    * 证书指纹：是整个证书内容的哈希值（如SHA-256），是证书的唯一身份证。
    * 公钥指纹：是证书中公钥部分的哈希值，是公钥的唯一标识。
  * 生成会话密钥：用证书中的 网站-公钥 加密 预主密钥（一段随机的内容和生成算法） 后发送给网站，网站用自己的 网站-私钥 解密，获得和客户端一样的 预主密钥和算法，两端生成会话密钥用于后续的通信。

###HTTPS中服务器发送给客户端的证书本身是明文的、未加密的。 证书如何保证安全？

步骤：证书发送（明文），会话密钥预主密钥（公钥加密），数据传输（使用会话密钥对称加密）

证书的安全性不依赖于隐藏其内容，而依赖于密码学上的**数字签名**机制。

1. **签发**：网站所有者向证书颁发机构（CA）申请证书。CA会严格验证申请者的身份（尤其是对EV扩展验证证书）。验证通过后，CA会使用自己的**私钥**（绝对保密）对证书的哈希值（指纹）进行加密，这个加密后的结果就是**数字签名**，并附加在证书上。

2. **传输**：当你的浏览器（客户端）连接一个HTTPS网站时，服务器会明文发送它的证书。

3. **验证**：你的浏览器收到证书后，会做以下几件事：
   
   * **a. 检查颁发者**：识别出是哪个CA签发的。
   
   * **b. 找到CA的公钥**：你的操作系统和浏览器都预装了一组受信任的CA的**公钥**（公钥是公开的）。
   
   * **c. 验证签名**：浏览器用CA的公钥去解密证书上的数字签名，得到CA当时计算的哈希值。同时，浏览器自己再根据收到的证书内容计算一次哈希值。
   
   * **d. 比对**：如果两个哈希值一致，就证明了两件事：
     
     * **完整性**：证书在传输过程中没有被篡改过（哪怕改一个字符，哈希值都会巨变）。
     
     * **真实性**：这个证书确实是由那个受信任的CA签发的，因为只有CA的私钥才能生成能被其公钥正确解密的签名。

**所以，攻击者虽然可以截获和看到证书的明文内容，但他无法伪造一个有效的证书。** 因为他没有CA的私钥，无法生成能被浏览器验证通过的有效签名。
