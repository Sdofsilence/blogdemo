# tcp/udp 实验
## TCP 相比 UDP 的关键功能/机制

1. 面向连接 (Connection-oriented)
   * 在传输数据前，必须通过“三次握手”建立连接，确保双方都准备好。
   * UDP 是无连接的，直接发包，不确认对方是否存在。
2. 可靠传输 (Reliability)
   * 确认应答 (ACK)：每个数据段都需要对方确认。
   * 超时重传 (Retransmission)：如果 ACK 没有按时收到，会重发。
   * 序列号 (Sequence Number)：保证数据按顺序到达，避免乱序。
   * 丢包检测：通过重复 ACK 或超时来发现丢包。
3. 有序传输 (In-order delivery)
   * TCP 会对乱序到达的数据进行重排，交付给应用层时是连续的字节流。
   * UDP 不保证顺序，应用层自己处理。
4. 字节流抽象 (Stream-oriented)
   * TCP 把应用层数据看作一个连续的字节流，不关心消息边界。
   * UDP 是“报文导向”的，应用层一次 sendto() 对应一次 recvfrom()。
5. 错误检测 (Checksum)
   * TCP 在首部和数据部分都有校验和，确保传输过程中数据未损坏。
   * UDP 也有校验和，但 TCP 的可靠性机制更完整。
6. 连接管理 (Connection management)
   * 三次握手：建立连接
   * 四次挥手：优雅关闭连接
   * 还支持 半关闭（一方关闭写，另一方还能读）。
7. KeepAlive（保活探测）
   * 你已经提到：用于检测长时间空闲的连接是否还存活。
8. 流量控制 (Flow Control)
   * 通过 滑动窗口 和 接收窗口 (rwnd)，防止发送方过快压垮接收方。
9. 拥塞控制 (Congestion Control)
   * 通过 慢启动、拥塞避免、快速重传、快速恢复 等算法，避免网络过载。
10. 半双工/全双工通信
    * TCP 连接是 全双工的，双方可以同时收发数据。
    * UDP 也能双向通信，但没有“连接”概念。

### 总结

* 所以，TCP 相比 UDP 的“最重要功能”可以归纳为：
  * 可靠性：确认、重传、序列号、乱序重排
  * 有序性：保证字节流顺序
  * 连接性：三次握手、四次挥手、KeepAlive
  * 控制机制：流量控制、拥塞控制、滑动窗口
  * 错误检测：校验和
  * 双工通信：全双工字节流

## 设计目标

* 基于 UDP：利用 UDP 的无连接、低开销特性。
* 可靠传输：保证数据不丢、不乱序。
* 简化实现：不必完全复刻 TCP，而是挑选最重要的功能。

### 必备功能模块

1. 连接管理（轻量版握手）
   * 三次握手简化版：
     * SYN(seq=x)
     * 服务端回复 SYN-ACK(seq=y, ack=x+1)
     * 客户端确认 ACK(ack=y+1)
   * 这样可以避免伪连接和半开连接问题。
   * 关闭连接时，可以用 单向 FIN（不必严格四次挥手）。

2. 序列号与确认机制
    * 每个数据包带 序列号 (seq)。
    * 接收方返回 确认号 (ack)。
    * 丢包检测：
      * 超时未收到 ACK → 重传
      * 或者收到重复 ACK → 快速重传

3. 滑动窗口（流量控制）
    * 发送方维护一个 发送窗口，接收方维护一个 接收窗口。

    * 接收方在 ACK 中携带 rwnd（接收窗口大小），告诉发送方自己还能接多少。

    * 避免接收方被压垮。

4. 拥塞控制（简化版）
不必实现 TCP 的全套算法（慢启动、拥塞避免、快速恢复）。

可以用一个 简单的自适应窗口：

初始窗口 = 1

每次成功 ACK → 窗口加倍（指数增长）

一旦丢包 → 窗口减半

这样能避免网络过载。

5. 保活机制（KeepAlive）
定期发送一个 心跳包（特殊标志位），确认连接仍然存活。

如果多次心跳无响应 → 认为连接断开。

6. 错误检测
UDP 已经有校验和，但你可以在自定义头部再加一个 CRC32，保证数据完整性。

### 报文格式（自定义头部）

Seq：序列号

Ack：确认号

Flags：SYN, ACK, FIN, RST, HEARTBEAT

Window：接收窗口大小

Checksum：额外校验

Payload：数据

### 简化实现思路

建立连接：三次握手（SYN → SYN-ACK → ACK）

传输数据：带序列号 + ACK + 滑动窗口

丢包处理：超时重传 + 快速重传

流量控制：接收方窗口通告

拥塞控制：简单加倍/减半算法

关闭连接：FIN → ACK

保活：定期心跳

### 总结
这个方案相当于做了一个 “迷你 TCP”：

保留了 可靠性、顺序性、流量控制、拥塞控制、保活

去掉了 复杂的状态机、Nagle 算法、延迟 ACK、全套拥塞算法

## 比 socket API 更底层 的方向,大致分三类

1. 用户态协议栈（自己实现 TCP/IP）

这些库绕过内核协议栈，在用户态直接处理数据包，常用于高性能网络或教学：

diy-TCP-IP-protocol：从零实现 TCP/IP 协议栈（ARP/IP/ICMP/UDP/TCP），C 语言写的教学项目

lwIP：轻量级 TCP/IP 协议栈，常用于嵌入式系统（路由器、IoT）

DPDK (Data Plane Development Kit)：Intel 主导的高性能用户态网络库，直接绕过内核驱动收发包

mTCP：基于 DPDK 的用户态 TCP 协议栈，专注高并发

2. 网络通信库（事件驱动 / 高性能 IO）
这些库不是协议栈本身，但提供了高性能的网络编程接口：

libevent：事件驱动网络库，支持 epoll/kqueue/select

Boost.Asio：现代 C++ 异步 IO 库，支持 TCP/UDP/串口/定时器

libuv：跨平台异步 IO 库（Node.js 底层就是它）

muduo：陈硕的 C++ 网络库，基于 Reactor 模型，教学和工程都很经典

3. 网络驱动 / 原始接口
如果你想操作 原始数据包 或写驱动：

Raw Socket：C 语言直接用 socket(AF_PACKET, SOCK_RAW, ...)（Linux），能收发二层帧

libpcap / WinPcap / npcap：抓包和注入数据包的库，常用于写 sniffer、防火墙

PF_RING：高性能抓包框架，比 libpcap 更快

Netmap：用户态高速收发包框架，适合写自定义协议栈

eBPF/XDP（Linux 内核扩展）：可以在内核里挂载小程序，做包过滤、转发、负载均衡

### 学习路径建议

入门：Raw Socket + libpcap，先能收发原始包

进阶：读 lwIP 或 diy-TCP-IP-protocol，理解协议栈实现

高性能：尝试 DPDK/mTCP/Netmap，体验用户态协议栈

驱动层：研究 Linux 内核 netdev 驱动，或用 eBPF/XDP 写内核扩展

### 总结

如果你想 学习协议栈原理 → 推荐 diy-TCP-IP-protocol、lwIP

如果你想 写高性能用户态网络 → 推荐 DPDK、mTCP、Netmap

如果你想 写驱动/抓包工具 → 推荐 libpcap、PF_RING、eBPF
