## 7.1 线程间划分任务(可以理解为多线程设计模式)

* 1.单纯的增加线程对互相不影响的数据划分执行相同的操作 2.递归划分给多线程(或者使用线程池搭配任务堆栈划分子任务) 3按任务类型划分：分为两种，一种是不同的线程执行不同的职责，另一种是流水线划分任务，每个线程负责一个阶段。
* 7.1.1 **在处理开始之前将数据分配到各个线程中**
  * 最简单的并行算法，就是并行化的  std::for_each  ，其会对一个数据集中每个元素执行同一个操作。最简单的分配方式：第一组N个元素分配一个线程，下一组N个元素再分配一个线程，以此类推。不管数据怎么分，每个线程都会对分配给它的元素进行操作，不过并不会和其他线程进行沟通，直到处理完成。
    * 有时，数据不能预先整齐地划分，因为只有在处理数据时，必要的划分才会变得明显。这在快速排序等递归算法中尤为明显；因此，他们需要一种不同的方法
* 7.1.2 **递归划分数据**
  * 在类似quick_sort排序算法中，每一级的递归都会多次调用quick_sort函数，因为需要知道哪些元素在中枢元素之前和之后。递归调用是完全独立的，因为其访问的是不同的数据集，并且每次迭代都能并发执行。使用std::async（）在每个阶段为较低的块生成异步任务。通过使用std::async()，您可以请求c++线程库来决定何时在新线程上实际运行任务以及何时同步运行任务。如果您正在对大量数据进行排序，那么为每个递归生成一个新线程将很快导致大量线程。性能方面，如果你有太多的线程，你可能会减慢应用程序的速度。如果数据集非常大，也有可能耗尽线程。
  * 一种替代方法是使用std::thread::hardware_concurrency（）函数来选择线程的数量。然后，不必为递归调用启动一个新线程，只需将要排序的块推入一个线程安全的堆栈(任务堆栈)，例如第5章和第6章中描述的堆栈之一。如果线程没有其他事情可做，要么是因为它已经完成了对所有块的处理，要么是因为它正在等待对一个块进行排序，那么它可以从堆栈中取出一个块并进行排序。
  * 这个方案中，不用为spawn_task产生的无数线程所困扰，也不用再依赖C++线程库，为选择执行线程的数量(就像  std::async()  那样)。该方案制约线程数量的值就是  std::thread::hardware_concurrency()  的值，这样就能避免任务过于频繁的切换。不过，这里还有两个问题：线程管理和线程通讯。要解决这两个问题就要增加代码的复杂程度，虽然线程对数据项是分开处理的，不过所有对栈的访问，都可以向栈添加新的数据块，并且移出数据块以作处理。重度竞争会降低性能(即使使用无锁(无阻塞)栈)，原因将会在后面提到。
  * 这个方案使用到了一个特殊的线程池——所有线程的任务都来源于一个等待链表，然后线程会去完成任务，完成任务后会再来链表提取任务。这个线程池很有问题(包括对工作链表的竞争)，这个问题的解决方案将在第8章提到。关于多处理器的问题，将会在本章后面的章节中做出更为详细的介绍(详见7.2.1)。
  * 在处理开始前对数据进行划分和递归地对数据进行划分都假定数据本身事先是固定的，而您只是在寻找划分的方法。情况并非总是如此；如果数据是动态生成的或来自外部输入，则此方法不起作用。在这种情况下，按任务类型划分工作可能比根据数据划分工作更有意义。
* 7.1.3 **按任务类型划分数据**
  * 通过将不同的数据块分配给不同的线程（无论是事先分配还是在处理过程中递归分配）来在线程之间分配工作，仍然基于这样一个假设：每个线程在处理每个数据块时都将执行几乎相同的任务。一个将工作进行分割的替代方法是将任务分配给不同的类型的线程，每个类型的线程负责完成一项特定的任务，就像水管工和电工在建造房屋时会执行不同的任务一样。这些不同类型的线程可能并不需要处理相同的数据，但如果他们确实处理相同的数据，那也是出于不同的目的。
    * 这就是通过将并发相关功能分离而形成的工作分工模式。每个线程被分配不同的任务，并独立执行这些任务，且不受其他线程的干扰。在某些情况下，其他线程可能会向其传递数据或触发需要处理的事件，但从整体上看，每个线程专注于完成单一职责。本质上，这体现了优秀的软件设计原则，即每段代码应仅承担一项明确的职责。
    * *按任务类型划分工作以分离关注点*
      * 当有多个任务需要在一段时间内连续运行，或者应用程序需要能够处理传入事件时，单线程应用程序必须使用单一职责原则来处理冲突（例如用户按键或传入的网络数据），即使在其他任务正在进行时也是如此。
        * 在单线程世界中，你最终需要手动编写代码，执行一点任务a，执行一点任务B，检查按键，检查传入的网络数据包，然后循环执行另一点任务a。这意味着任务a的代码最终会变得复杂，因为需要保存其状态并定期将控制返回给主循环。如果在循环中添加了太多任务，那么速度可能会慢得太多，用户可能会发现对按键的响应时间太长。
        * 这就是线程的用武之地。如果您在单独的线程中运行每个任务，操作系统将为您处理这些任务。在任务A的代码中，您可以专注于执行任务，而不必担心保存状态和返回主循环，也不必担心在此之前花费多长时间。操作系统将自动保存状态，并在适当的时候切换到任务B或C，如果目标系统有多个内核或处理器，任务A和任务B可能能够真正并发地运行。处理按键或网络数据包的代码现在将及时运行，每个人都是受益者：用户得到了及时的响应，作为开发人员的您拥有了更简单的代码，因为每个线程都可以专注于与其职责直接相关的操作，而不是与控制流和用户交互混淆在一起。
        * 这听起来像是一个美好而乐观的愿景。真的会是这样吗？就像所有事情一样，这取决于细节。如果一切真的都是独立的，而且这些线程之间无需相互交流，那么确实会如此简单。不幸的是，现实世界很少如此。这些后台任务通常在执行用户请求的操作，它们需要以某种方式更新用户界面来告知用户任务已完成。或者，用户可能想要取消任务，这就需要用户界面以某种方式向后台任务发送消息告知其停止。这两种情况都需要仔细思考和设计以及适当的同步，但关注点仍然是分开的。用户界面线程仍然只处理用户界面，但当其他线程要求时，它可能需要进行更新。同样，运行后台任务的线程仍然只专注于完成该任务所需的操作；只是其中一项操作变成了“允许被另一个线程停止”。在任何一种情况下，线程之间都没有直接交互。他们只关心请求是针对他们的，并且直接与他们的职责相关，而不关心请求来自何处。
        * 使用多个线程分离关注点有两个主要危险。首先，您可能会分离错误的关注点。需要检查的症状包括线程之间有大量数据共享，或者不同线程最终相互等待；这两种情况都归结为线程之间的过度通信。如果发生这种情况，值得审视通信的原因。如果所有通信都与同一个问题相关，那么这有可能应该是单个线程的主要责任，并从所有引用该问题的线程中提取出来。另一方面，如果两个线程之间的通信非常频繁，而与其他线程的通信相对较少，那么也许它们应该合并为一个线程。
        * 在按任务类型将工作分配到线程时，您并不必局限于完全隔离的情况。如果多个输入数据集需要应用相同的操作序列，您可以将工作划分，使每个线程执行总体序列中的一个阶段(下面的任务序列)。
    * *在多个线程之间划分任务序列*
      * 如果您的任务是对许多独立的数据项应用相同的操作序列，则可以使用管道来利用系统的可用并发性。这类似于物理管道：数据从一端流入，通过一系列操作（管道），然后从另一端流出。
      * 通过这种方式分配工作，您为管道中的每个阶段创建一个单独的线程——每个操作对应一个线程。当操作完成时，数据元素会被放置到队列中，以便被下一个线程拾取。这使得执行序列中第一个操作的线程能够在管道中的第二个线程处理第一个元素时，开始处理下一个数据元素。
      * 这是在线程之间划分数据的一种替代方法，如第7.1.1节所述，并且适用于在操作开始时输入数据本身并不完全已知的情况。例如，数据可能通过网络传入，或者序列中的第一个操作可能是扫描文件系统以确定要处理的文件。
      * 管道技术在每个操作都耗时的情况下也很有效;通过在线程之间而不是在数据之间划分任务，您可以改变性能特征。按序列处理(而不是划分)，假设每一阶段3秒，四个阶段，四个线程，当管道就绪，您每 3 秒处理一个项目，而不是每 12 秒处理一次四个项目。
      * 处理整个批次所需的总时间更长，因为在最后一个核心开始处理第一个项目之前，您需要等待 9 秒。但在某些情况下，更顺畅、更规律的处理可能是有益的。例如，考虑一个用于观看高清数字视频的系统。为了使视频可观看，通常需要至少每秒 25 帧，理想情况下更多。此外，观看者需要这些帧均匀间隔，以给予连续运动的印象；如果一个应用程序可以以每秒解码 100 帧，但在暂停一秒钟后才显示 100 帧，然后再暂停一秒钟，再显示 另外 100 帧，那么这种应用也无济于事。另一方面，观众在开始观看视频时可能会乐于接受几秒钟的延迟。在这种情况下，利用以稳定速率输出帧的管道进行并行处理可能更为可取。

## 7.2 影响并发代码的因素

* 如果您使用并发是为了在具有多个处理器的系统上提高代码的性能，那么您需要知道哪些因素会影响性能。即使您只是使用多个线程来分离关注点，也需要确保这不会对性能产生不利影响。
* 7.2.1 **处理器的数量**
  * 处理器个数是影响多线程应用的首要因素。初步估计，一个单核16核心处理器相当于4个四核处理器或16个单核处理器：在这三种情况下，系统均可以同时运行16个线程。如果您想利用这一点，您的应用程序必须至少有16个线程。如果少于16个，您就会浪费处理器的性能（除非系统同时运行其他应用程序，但我们暂时忽略这种可能性）。另一方面，如果您有超过16个线程实际上已经准备好运行（并且没有被阻塞，等待某些操作），您的应用程序将在线程之间切换，从而浪费处理器时间，如第一章所述。当这种情况发生时，称之为过度订阅。为了使应用程序能够根据硬件能够并发运行的线程数量调整线程的数量，C++ 11标准线程库提供了std::thread::hardware_concurrency()。
  * 直接使用 std::thread::hardware_concurrency() 需要谨慎；您的代码并未考虑系统中正在运行的其他线程，除非您明确共享该信息。在最坏的情况下，如果多个线程同时调用使用 std::thread::hardware_concurrency() 进行扩展的函数，将会导致严重的超额订阅。std::async() 避免了这个问题，因为该库了解所有的调用并能够进行适当的调度。谨慎使用线程池也可以避免这个问题。
  * 不过，即使已经考虑到所有在应用中运行的线程，程序还要被同时运行的其他程序所影响。虽然系统能提供选择线程数量的机制，但这种机制已经超出C++标准的范围。
  * 这种情况的另一个转折是，问题的理想算法可以取决于问题的大小与处理单元的数量相比。如果您有一个具有许多处理单元的大规模并行系统，总体上执行更多作的算法可能比执行较少作的算法完成得更快，因为每个处理器只执行少量作(并行化的算法比非并行化的算法多执行但是更快)。
* 7.2.2 **数据竞争和缓存乒乓**
  * (数据竞争)随着处理器数量的增加，出现另一个问题的可能性和性能影响也在增加：多个处理器试图访问相同的数据。当两个线程并发的在不同处理器上执行，并且对同一数据进行读取，通常不会出现问题；因为数据将会拷贝到每个线程的缓存中，并且让两个处理器同时进行处理。不过，当有线程对数据进行修改的时候，这个修改需要更新到其他核芯的缓存中去，就要耗费一定的时间。根据线程的操作性质，以及使用到的内存序，这样的修改可能会让第二个处理器停下来，等待硬件内存更新缓存中的数据。即便是精确的时间取决于硬件的物理结构，不过根据CPU指令，这是一个特别特别慢的操作，相当于执行成百上千个独立指令。
  * 下面代码(循环+独占操作)，如果do_something()足够短，或有很多处理器来对这段代码进行处理时，处理器将会互相等待；一个处理器准备更新这个值，另一个处理器正在修改这个值，所以该处理器就需要等待第二个处理器更新完成，*并且完成更新传递时才能执行更新*。这种情况被称为高竞争(high contention)。

    ```cpp
        std::atomic<unsigned long> counter(0);
        void processing_loop()
        {
                while(counter.fetch_add(1,std::memory_order_relaxed)<100000000) {
                    do_something(); }
        }
    ```

  * (乒乓缓存)当多个CPU核心频繁修改同一缓存行（Cache Line） 的数据时，引发缓存一致性协议（如MESI）的协调操作。只要有核心修改数据，每次修改需同步给其他核心的缓存，其他需要读取数据的核心就需要等待缓存同步。如果多个核心轮流写入和读取，那么缓存就一直在多个核心之间反复传输。这种情况会对应用的性能有着重大的影响。
  * 您可能认为这种情况不会发生在您身上；毕竟，您没有这样的循环。您确定吗？那么互斥锁呢？如果您在循环中获取一个互斥锁，从数据访问的角度来看，您的代码与之前的代码类似。为了锁定互斥锁，另一个线程必须将构成互斥锁的数据传送到其处理器，并对其进行修改。当完成后，它再次修改互斥锁以解锁，并且互斥锁数据必须传送给下一个线程以获取互斥锁。这种数据传送时间还需要加上第二个线程必须等待第一个线程释放互斥锁的时间。

    ```cpp
    std::mutex m;
    my_data data;
    void processing_loop_with_mutex(){
        while(true)
        {        std::lock_guard<std::mutex> lk(m);
                if(done_processing(data)) break;
        }
    }
    ```

  * 由于互斥锁的竞争效果通常与原子操作的竞争效果不同，原因在于互斥锁的使用自然地在操作系统级别而不是在处理器级别上序列化线程。如果有足够多的线程准备好运行，操作系统可以在一个线程等待互斥锁时调度另一个线程运行，而处理器停滞会阻止该处理器上任何线程的运行。然而，它仍然会影响那些竞争互斥锁的线程的性能；毕竟，它们一次只能运行一个。
  * 回顾第2章，一个很少更新的数据结构可以被一个“单作者，多读者”互斥量(详见2.3.2)保护。乒乓缓存效应可以抵消互斥所带来的收益(工作量不利时)，因为所有线程访问数据(即使是读者线程)都会对互斥量进行修改。随着处理器对数据的访问次数增加，对于互斥量的竞争就会增加，并且持有互斥量的缓存行将会在核芯中进行转移，因此会增加不良的锁获取和释放次数。有些方法可以改善这个问题，就是让互斥量对多行缓存进行保护，不过这样的互斥量需要自己去实现。
  * 如果乒乓缓存是一个糟糕的现象，该怎么避免它呢？本章最后，答案会与提高并发潜能的指导意见相结合：减少两个线程对同一个内存位置的竞争。虽然，要实现起来并不简单。即使给定内存位置被一个线程所访问，可能还是会有乒乓缓存存在，是因为另一种叫做伪共享(false sharing)的效应。

* 7.2.3 **伪共享(false sharing)**
  * 处理器缓存通常不会用来处理单个存储点，会用来处理称为缓存行(cache lines)的内存块。内存块通常大小为32或64字节，实际大小需要由正在使用的处理器模型来决定。因为硬件缓存确定处理缓存行大小的内存块，较小的数据项就在同一内存行的相邻内存位置上。有时这样的设定还是挺不错：当线程访问的一组数据是在同一缓存行中，对于应用的性能来说就要好于对多个缓存行进行传播。不过，在同一缓存行存储的是无关数据时，且需要被不同线程访问，这就会造成性能问题。
  * 假设你有一个整型值的数组和一组线程，它们每个线程都访问数组中的自己的条目，并且会重复执行这一操作，包括更新。由于整型通常比缓存行小得多，因此这些数组条目中的很多会在同一个缓存行中。因此，即使每个线程只访问自己的数组条目，缓存硬件仍然需要进行缓存乒乓。当访问条目0的线程需要更新值时，缓存行的所有权需要转移到运行该线程的处理器，而在那个线程需要更新其数据项时，又必须转移到运行条目1线程的缓存。这种情况下，缓存行是共享的，尽管数据本身并不共享，因此称为虚假共享。这里的解决方案是将数据结构化，以便由同一线程访问的数据项在内存中彼此靠近（因此更可能在同一个缓存行中），而由不同线程访问的数据项则在内存中相距较远，因此更可能在不同的缓存行中。你将在本章后面看到这如何影响代码和数据的设计。
  * 如果多个线程访问来自同一缓存行的数据是有害的，那么单个线程访问的数据的内存布局会如何影响情况？

* 7.2.4 **你的数据有多接近(时间/空间局部性原理)**
  * 而虚假共享是由于一个线程访问的数据过于接近另一个线程访问的数据而导致的，另一个与数据布局相关的陷阱直接影响单个线程的性能。问题在于数据的接近性：如果一个线程访问的数据在内存中分散开来，那么它很可能位于不同的缓存行中。另一方面，如果一个线程访问的数据在内存中靠近在一起，那么它更可能位于同一缓存行中。因此，如果数据分散开，就必须从内存加载更多的缓存行到处理器缓存中，这可能会增加内存访问延迟，使性能低于相比于数据较为接近的情况。
  * 此外，如果数据分散，当前线程所需的数据所在的缓存行也可能包含不是当前线程所需的数据。这种极端情况下，缓存中的无关数据可能会比相关数据还多。这会浪费宝贵的缓存空间，从而增加处理器出现缓存缺失的机会，导致需要从主内存中获取数据项，即使该数据项曾经在缓存中，因为它必须将该项从缓存中移除以为其他数据腾出空间。
  * 这在单线程代码中很重要，那么我为什么在这里提到它呢？原因是任务切换。如果系统中的线程数量超过了核心数量，每个核心将会运行多个线程。这增加了缓存的压力，因为你试图确保不同的线程访问不同的缓存行，以避免虚假共享。因此，当处理器切换线程时，如果每个线程使用的数据分散在多个缓存行中，而不是每个线程的数据紧靠在同一个缓存行中，那么它更可能需要重新加载缓存行。
  * 如果线程的数量超过了核心或处理器的数量，操作系统可能会选择在一个核心上调度一个线程进行一个时间片，然后在下一个时间片上调度到另一个核心。因此，这将需要将该线程数据的缓存行从第一个核心的缓存转移到第二个核心的缓存；需要转移的缓存行越多，所需的时间就越长。尽管操作系统通常会在可能的情况下避免这种情况，但这种情况确实会发生，并且在发生时会影响性能。
  * 当许多线程准备运行而不是等待时，任务切换问题特别普遍。这是我们已经提到过的问题：超额订阅。

* 7.2.5 **过度订阅和过多的任务切换**
  * 在多线程系统中，通常会有比处理器更多的线程，除非你是在大规模并行硬件上运行。然而，线程往往会花时间等待外部I/O完成，或者在互斥锁上被阻塞，或者等待条件变量等，因此这并不是问题。拥有额外的线程使得应用程序能够执行有用的工作，而不是让处理器在线程等待时闲置。
  * 这并不总是一件好事。如果你有太多额外的线程，将会有更多准备运行的线程比可用的处理器还要多，操作系统将不得不开始频繁地进行任务切换，以确保它们都能获得公平的时间片。正如你在第一章中看到的，这可能会增加任务切换的开销，并加重由于缺乏接近而导致的任何缓存问题。当你有一个任务不断创建新的线程而没有限制时，就会出现过度订阅的情况，就像第四章的递归快速排序一样，或者当你根据任务类型分离的自然线程数量超过处理器数量，并且工作是自然的CPU绑定(cpu密集型任务)而不是I/O绑定(i/o密集型任务)时也会出现这种情况。对于 CPU 密集型任务，应尽量控制并发线程数 ≈ CPU 核心数。对于 I/O 密集型任务，可以适当增加线程数，因为很多线程处于等待状态。
  * 如果由于数据划分而产生的线程过多，您可以限制工作线程的数量，就像您在第8.1.2节中看到的那样。如果超额订阅是由于工作自然划分造成的，那么您能做的事情就不多了，除非选择不同的划分。在这种情况下，选择适当的划分可能需要比您现有的知识更多的目标平台知识，只有在性能不可接受并且可以证明改变工作划分确实能改善性能时，才值得这样做。
  * 其他因素也会影响多线程代码的性能。（例如，即使两个单核处理器和一个双核处理器是同一种 CPU 类型且时钟速度相同，缓存乒乓操作的成本也可能有很大差异）但这些(7.2.1~7.2.5)是会对性能产生显著影响的主要因素。现在让我们看看这如何影响代码和数据结构的设计。

## 7.3 为多线程‘性能’设计数据结构

* 在第7.1节中，我们探讨了在线程之间分配工作的各种方法，而在第7.2节中，我们研究了可能影响代码性能的各种因素。当设计多线程性能的数据结构时，您如何利用这些信息？这个问题与第5章和第6章讨论的安全并发访问数据结构的设计问题不同。
* 在为多线程性能设计数据结构时，需要牢记的关键要素包括争用、虚假共享和数据接近性。这三者都可能对性能产生重大影响，您通常只需通过更改数据布局或更改哪些数据元素分配给哪个线程就可以改善性能。首先，让我们来看看一个简单的改进：在线程之间分配数组元素。
* 7.3.1 为复杂操作分隔‘数组’元素
  * 假设你有一些偏数学的计算任务，比如：需要将两个很大的矩阵进行相乘。现在，假设两个矩阵都有上千行和上千列，使用多线程来优化矩阵乘法。通常，非稀疏矩阵可以用一个大数组来代表，也就是第二行的元素紧随着第一行的，以此类推。为了完成矩阵乘法，这里就需要三个大数组。为了优化性能，需要仔细考虑数据访问的模式，特别是向第三个数组中写入的方式。
  * 在线程之间划分工作的方法有很多。假设您拥有比可用处理器更多的行/列，您可以让每个线程计算结果矩阵中若干列的值，或者让每个线程计算若干行的结果，甚至让每个线程计算矩阵的一个子矩形的结果
  * 回顾一下7.2.3和7.2.4节，对于一个数组来说访问连续的元素是最好的方式，因为这将会减少缓存的刷新，并且降低伪共享的概率。如果要让每个线程处理几行，线程需要读取第一个矩阵中的每一个元素，并且读取第二个矩阵上的相关行上的数据，不过这里只需要对列的值进行写入。给定的两个矩阵是以行连续的方式存储，这就意味着当访问第一个矩阵的第一行的前N个元素，而后是第二行的前N个元素，以此类推(N是列的数量)。其他线程会访问每行的的其他元素，访问相邻的列，所以从行上读取的N个元素也是连续的，这将最大程度的降低伪共享的几率。当然，如果空间已经被N个元素所占有，且N个元素也就是每个缓存行上具体的存储元素数量，就会让伪共享的情况消失，因为线程将会对独立缓存行上的数据进行操作。
