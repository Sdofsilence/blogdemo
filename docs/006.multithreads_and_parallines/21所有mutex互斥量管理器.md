# c++管理互斥量（mutex）的 RAII 包装器
1. std::lock_guard
* 特性：
    * 最简单的锁管理类。
    * 构造时立即锁定互斥量，析构时自动释放。
    * 不支持手动解锁或重新锁定。

* 适用场景：
    * 保护临界区代码块，无需复杂锁操作。
    * 确保在作用域退出时锁被释放。

2. std::unique_lock
* 特性：
    * 功能最灵活的锁管理类。
    * 支持 延迟加锁、手动解锁/重新锁定、转移所有权。
    * 可配合条件变量（std::condition_variable）使用。

* 适用场景：
    * 需要条件变量的等待操作（如 cv.wait(lock)）。
    * 需要手动控制锁的生命周期（如分段加锁）。

3. std::shared_lock
* 特性：
    * 专为 共享互斥量（std::shared_mutex）设计。
    * 允许多个线程同时获取 共享锁（读锁），但独占锁（写锁）只能被一个线程持有。

* 适用场景：
    * 读多写少的场景（读写锁优化）。
    * 需要并发读取共享数据，但写入时互斥。

4. std::scoped_lock（C++17 引入）
* 特性：
    * 用于 同时锁定多个互斥量，避免死锁（使用死锁避免算法）。
    * 类似 std::lock_guard，但支持多个互斥量。

* 适用场景：
    * 需要同时获取多个互斥量。
    * 替代 std::lock + std::lock_guard 的组合。

# 对比总结
| 特性 | lock_gard | unique_gard | shared_gard | scoped_gard |
|:---|:---|:---|:---|:---|
| 手动解锁    | N | Y | Y | N |
| 延迟加锁    | N | Y | Y | N |
| 配合条件变量 | N | Y | N | N |
| 支持共享锁   | N | N | Y | N |
| 支持多互斥量 | N | N | N | Y |
| 所有权转移   | N | Y| N | N |
| c++版本     | C++11 | C++11| C+++14 | C++17 |

# 关键选择建议
1. 简单临界区 → lock_guard。
2. 需要条件变量或灵活控制 → unique_lock。
3. 读写锁优化（读多写少） → shared_lock + shared_mutex。
4. 同时锁定多个互斥量 → scoped_lock。

# 注意事项
* 避免死锁：使用 scoped_lock 锁定多个互斥量时，确保所有线程以相同顺序加锁。
* 锁粒度控制：尽量缩小锁的作用域，减少线程争用。
* 性能开销：unique_lock 比 lock_guard 稍重（因功能更多），但大多数场景可忽略。
