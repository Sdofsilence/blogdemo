# 4c++内存模型与原子类型
## 4.1 c++11内存模型
* 4.1.1 对象和内存位置.这里有四个需要牢记的原则：
    1.  每一个变量都是一个对象，包括作为其成员变量的对象。
    2.  每个对象至少占有一个内存位置。
    3.  基本类型都有确定的内存位置(无论类型大小如何，即使他们是相邻的，或是数组的一部分)。
    4.  相邻位域是相同内存中的一部分。
* 4.1.2 对象、内存位置和并发
    * 部分对于C++的多线程来说是至关重要的：所有东西都在内存中。当两个线程访问不同的内存位置时，不会存在任何问题，一切都工作顺利。当两个线程访问同一个内存位置，就要小心了。如果没有线程更新数据，那还好；只读数据不需要保护或同步。当有线程对内存位置上的数据进行修改，那就有可能会产生条件竞争。
    * 为了避免条件竞争，两个线程就需要一定的执行顺序。第一种方式，使用互斥量来确定访问的顺序；当同一互斥量在两个线程同时访问前被锁住，那么在同一时间内就只有一个线程能够访问到对应的内存位置，所以后一个访问必须在前一个访问之后。另一种是使用原子操作(详见第2节中对于原子操作的定义)，决定两个线程的访问顺序。使用原子操作来规定顺序在第3节中介绍。当多于两个线程访问同一个内存地址时，对每个访问这都需要定义一个顺序。
    * 另一个重点是：当程序对同一内存地址中的数据访问存在竞争，可以使用原子操作来避免未定义行为。当然，这不会影响竞争的产生——原子操作并没有指定访问顺序——但原子操作把程序拉回到定义行为的区域内。在了解原子操作前，还有一个有关对象和内存地址的概念需要重点了解：修改顺序。
* 4.1.3 修改顺序(见4.3.1 *Modification order)
    * 是指对于某个特定对象（比如一个变量），它的所有写操作（包括初始化）按照时间顺序排列起来的一个序列。这个顺序要求是全局一致的，所有线程必须看到相同的顺序，否则导致数据竞争，从而引发未定义行为。
        * 对于非原子变量（如普通的 int、struct 类型等），你必须自己通过锁、内存屏障或其他同步手段来确保所有线程看到一致的修改顺序，否则就会导致数据竞争，从而引发未定义行为。
        * 对于原子变量（如 std::atomic<T>），编译器和 CPU 会自动插入适当的同步指令，以确保所有线程都能看到一致的修改顺序。
    * 修改顺序的一致性要求。要求修改顺序是全局一致的，所有线程必须看到相同的顺序。
    * 不同对象间的顺序可以不一致。虽然每个对象的修改顺序是全局一致的，但不同对象之间的操作顺序可以在不同线程中有不同的观察结果。

## 4.2 原子类型
* 原子操作是不可分割的操作。你不能从系统中的任何线程观察到这样的操作在半完成状态；它要么已经完成，要么没有完成。如果读取对象值的加载操作是原子的，并且对该对象的所有修改也是原子的，那么该加载将检索到对象的初始值或由其中一次修改存储的值。
* 非原子操作可能会被另一个线程观察到只完成一半。如果这个操作是一个存储操作，那么其他线程看到的值，可能既不是存储前的值，也不是存储的值，而是别的什么值。如果非原子操作是一个加载操作，它可能先取到对象的一部分，然后值被另一个线程修改，然后它再取到剩余的部分，所以它取到的既不是第一个值，也不是第二个值，而是两个值的某种组合。
* 4.2.1 std::atomic类型
    * 标准原子类型定义在头文件 <atomic> 中。这些类型的所有操作都是原子的，语言定义中只有这些类型的操作是原子的，不过可以用互斥锁来模拟原子操作。实际上，标准原子类型的实现就可能是这样模拟出来的：它们(几乎)都有一个 is_lock_free() 成员函数，这个函数可以让用户查询某原子类型的操作是直接用的原子指令(x.is_lock_free() 返回 true )，还是内部用了一个锁结构( x.is_lock_free() 返回 false )。
    * 唯一不提供 is_lock_free() 成员函数的类型是 std::atomic_flag。该类型是一个非常简单的布尔标志，它保证是无锁的。一旦拥有一个简单的无锁布尔标志，就可以用它实现一个简单的锁，因此可以以此为基础实现所有其他原子类型。std::atomic_flag 类型的对象初始化为清除状态，然后可以进行查询和设置（使用 test_and_set() 成员函数）或清除（使用 clear() 成员函数）。除此之外没有其他任何操作(c++20添加了wait和test两个函数)。
    * 剩余的原子类型都是通过 std::atomic<> 类模板的特化进行访问的，这些类型功能更为全面，但可能不是无锁的。除了直接使用 std::atomic<> 类模板之外，您还可以使用 atomic_[type] 这样的名称集合来引用实现提供的原子类型。不要在同一程序中将这些替代名称与直接命名 std::atomic<> 特化混合使用，可能会导致不可移植的代码。除了基本的原子类型外，C++标准库还提供了一组与各种非原子标准库类型（如std::size_t atomic_size_t）相对应的原子类型的typedef。一般来说，针对您希望使用的 T，直接使用 std::atomic<T> 更为简单，而不是使用替代名称。
    * 标准原子类型在常规意义上是不可复制或不可赋值的，因为它们没有复制构造函数或复制赋值运算符。然而，它们确实支持对相应内置类型的赋值和隐式转换，以及直接的 load() 和 store() 成员函数，exchange()、compare_exchange_weak() 和 compare_exchange_strong()。它们还支持复合赋值运算符（在适当的情况下）：+=、-=、*=、|= 等，整型和 std::atomic<> 指针的专用化支持 ++ 和 --。这些操作分为三个类别：
        * 1.Store操作，可选如下顺序：memory_order_relaxed, memory_order_release,memory_order_seq_cst。
        * 2.Load操作，可选如下顺序：memory_order_relaxed, memory_order_consume,memory_order_acquire, memory_order_seq_cst。
        * 3.Read-modify-write(读-改-写)操作，可选如下顺序：memory_order_relaxed,memory_order_consume, memory_order_acquire, memory_order_release,memory_order_acq_rel, memory_order_seq_cst。

* 4.2.7非成员的原子操作函数
    * 对原子类型操作，不同的原子类型中也有等价的非成员函数存在。大多数非成员函数的命名与对应成员函数有关，需要“  atomic_”作为前缀(比如，std::atomic_load())。它们有两种变体：一种是不带标签，另一种带有显式后缀和一个或多个用于内存排序的附加参数（例如，std::atomic_store(&atomic_var,new_value) 与 std::atomic_store_explicit(&atomic_var,new_value,std::memory_order_release)）。而通过成员函数引用的原子对象是隐式的，所有自由函数都将指向原子对象的指针作为第一个参数。a.load(std::memory_order_acquire) 的等效函数是 std::atomic_load_explicit(&a, std::memory_order_acquire)。这些非成员函数被设计为与 C 语言兼容，因此在所有情况下它们使用指针而不是引用。例如，compare_exchange_weak() 和 compare_exchange_strong() 成员函数的第一个参数（期望值）是引用，而 std::atomic_compare_exchange_weak() 的第二个参数（第一个是对象指针）是指针。std::atomic_compare_exchange_weak_explicit() 还要求指定成功和失败的内存顺序，而 compare/exchange 成员函数同时具有单一的内存顺序形式（默认为 std::memory_order_seq_cst）和一个分别接受成功和失败内存顺序的重载版本。
    * C++标准库还提供了以原子方式访问std::shared_ptr<>实例的非成员函数。这有悖于只有原子类型支持原子操作的原则，因为std::shared_ptr<>显然并不是原子类型。然而，C++标准委员会认为提供这些额外函数是相当重要的。可用的原子操作包括load、store、exchange和compare_exchange，这些操作作为对标准原子类型上相同操作的重载，接受std::shared_ptr<>*作为第一个参数。c++20后被std::atomic<std::shared_ptr>模版特化取代而弃用。与其他类型的原子操作一样，也提供了 _explicit 变体，以允许您指定所需的内存顺序，并且可以使用 std::atomic_is_lock_free() 函数来检查实现是否使用锁来确保原子性。
        > std::shared_ptr<my_data> p; \
        > std::atomic_load(&p); \
        > std::shared_ptr<my_data> local(new my_data); \
        > std::atomic_store(&p,local);

## 4.3 同步操作并强制执行顺序
* 原文介绍了一个用atomic操作(默认全局顺序一致性)来同步非atomic操作的例子。(其中a是非atomic，b是atomic)即一个线程中 write a happens-before b,另一个线程中read b happens-before a,因为默认内存序对同一个变量的读写是全局一致的(同步的)，所以b的write synchronizes-with the read,从而创造了一个 b 的write happens-before b的 read关系，happens-before关系具有传递性，所以有：a的write->b的write->b的read->a的read 这样的强制顺序。接下来详细介绍happens-before的 synchronizes-with两种关系。
* 4.3.1 synchronizes-with 关系
    * “同步发生”只能在原子类型之间进行。从根本上说，这种关系只能来源于对原子类型的操作。
    * `表达式求值：包括值计算（value computation）和副作用（side effect）。
        * 值计算：计算表达式返回的值。这可能涉及确定对象的标识（左值求值，例如，如果表达式返回对某个对象的引用）或读取先前赋给对象的值（纯右值求值，例如，如果表达式返回数字或某些其他值）。
        * 副作用的启动：访问（读取或写入）由 volatile 左值指定的对象，修改（写入）对象，调用库 I/O 函数，或调用执行任何这些操作的函数。
    * Sequenced-before(先序于):先序于是*同一线程内*求值 A 和 B 之间的非对称、传递、成对关系。如果 A 先序于 B（或等效地，B 后序于 A），则 A 的求值将在 B 的求值开始之前完成。(不同于as-if顺序原则(可以乱序只要不影响结果)，sequenced-before是一种强制要求)
    * 携带依赖：*在同一线程内*，求值A sequenced-before 求值B，同时也可能将依赖携带到B中（即，B依赖于A），如果以下任何一项为真(1.A 的值用作 B 的操作数 2.A 写入标量对象 M，B 从 M 读取 3.A 将依赖携带到另一个求值 X，并且 X 将依赖携带到 B)
    * Modification order：对任何特定原子变量的所有修改都是按照该原子变量特有的总顺序进行的。 所有原子操作都必须满足以下四个要求：
        * 1.写-写一致性：如果修改某个原子 M 的求值 A（写入）先序发生于修改 M 的求值 B，则 A 在 M 的修改顺序中出现在 B 之前。(A...B)解释：如果一个写操作 A 在程序顺序上早于另一个写操作 B（比如它们在同一个线程里按顺序执行），那么在修改顺序中，A 也必须排在 B 前面。
        * 2.读-读一致性：如果对某个原子 M 的值计算 A（读取）先序发生于对 M 的值计算 B，并且如果 A 的值来自对 M 的写入 X，则 B 的值是 X 存储的值，或者是对 M 的副作用 Y 存储的值，而 Y 在 M 的修改顺序中出现在 X 之后。(X...Y) 解释：如果一个线程连续两次读取一个原子变量，并且第一次读到了某个写入 X 的值，那么第二次读要么看到 X 的值，要么看到比 X 更新的值（即修改顺序中 X 之后的写入 Y）。
        * 3.读-写一致性：如果对某个原子 M 的值计算 A（读取）先序发生于对 M 的操作 B（写入），则 A 的值来自在 M 的修改顺序中早于 B 出现的副作用（写入）X。(X...B)解释：如果一个线程先读了一个值，然后写了一个新值，那么它读到的值一定来自修改顺序中比那个新值更早的写入。
        * 4.写-读一致性：如果对原子对象 M 的副作用（写入）X 先序发生于对 M 的值计算（读取）B，则求值 B 应从 X 或在 M 的修改顺序中跟随 X 的副作用 Y 中获取其值。(X...Y)解释：如果一个写操作 X 发生在一个读操作 B 之前，那么 B 必须看到 X 的值或比 X 更新的值。
    * Release sequence：在原子对象 M 上执行释放操作 A 之后，M 的 modification order (修改序列)的最长连续子序列，该子序列由以下各项组成：1由执行 A 的同一线程执行的写入。(直至 C++20) 2 由任何线程对 M 进行的原子读-修改-写操作。被称为以 A 为首的释放序列。重点是最长连续子序列由自己后续的写操作+其他线程的读-改-写操作组成(不能被中断,即不满足以上两个条件的写入操作，任意线程的读取操作不会中断释放序列，他只是不属于释放序列，也就是说释放序列和读取操作同步时，可能不是第一个遇到的读取操作，他和后续的读取操作都可以同步)。(释放的含义就是写操作，所以释放序列就是最长连续写序列).释放序列（Release Sequence）是C++内存模型中一个关键概念，它解决了非顺序一致（non-SC）内存模型下的synchronizes-with传递问题或(非常重要：因为他是同步 读-写-》读-写-》读-写》(每一个读写同步),或者同一线程中的写-》写-》写 对 其他线程的读同步)。
    * synchronizes-with：如果线程 A 中的原子存储是释放操作，线程 B 中从同一变量进行的原子加载是获取操作，并且线程 B 中的加载读取了线程 A 中的存储写入的值，则线程 A 中的存储同步于线程 B 中的加载。(保证可以看到A之前的所有写入操作(包括非原子的))
    * 依赖顺序先于(不用看消费操作不建议用)：在线程之间，如果以下任何一项为真，则求值 A 依赖顺序先于求值 B：1 A对某个原子M执行释放操作，并且在不同的线程中，B对同一原子M执行*消费操作*，并且B读取了由以A为首的释放序列的任何部分写入的值 2 A 依赖顺序先于 X，并且 X 将依赖携带到 B。
* 4.3.2 happens-before 关系
    * happens-before:无论线程如何，如果以下任何一项为真，则求值 A happens-before 求值 B：1 A Sequenced-before(同一线程中) B。2 A 线程间happens-before B。(c++20和c++26对传递性作出修改A->X->B)
    * 线程间happens-before：在线程之间，如果以下任何一项为真，则求值 A 线程间先序发生于求值 B：1 A 同步于 B。2 A 依赖顺序先于 B。3 A 同步于某个求值 X，并且 X 先序发生于 B。4 A 先序发生于某个求值 X，并且 X 线程间先序发生于 B。5 A 线程间先序发生于某个求值 X，并且 X 线程间先序发生于 B。
    * 可见的副作用(副作用：访问（读取或写入）由 volatile 左值指定的对象，修改（写入）对象):标量 M 上的副作用 A（写入）相对于 M 上的值计算 B（读取）是可见的，如果以下两者都为真：1 A happens-before B。2 没有其他对 M 的副作用 X，其中 A happens-before X 并且 X happens-before B。如果副作用 A 相对于值计算 B 是可见的，那么在 M 的修改顺序的最长连续副作用子集中， B 不会在它之前发生(也就是说A是最后一个副作用，且B在它之后发生)，称为可见的副作用序列（由 B 确定的 M 值将是这些副作用之一存储的值）。(即保证了B读取的值是A写入的值，而不是中间副作用X写入的值，可见就是能读取到它写入的值)
    * 注意：线程间同步可归结为防止数据竞赛（通过建立 happens-before 关系）和定义哪些副作用在什么条件下可见。(见线程间同步理解.md)
* 4.3.3 atomic操作的memory ordering
    * 三类操作
        * 1.具有 memory_order_consume 或更强内存顺序的原子加载是消费操作(不建议使用)。
        * 2.具有 memory_order_acquire 或更强内存顺序的原子加载是*获取操作*。互斥量上的 lock() 操作也是获取操作。
        * 3.具有 memory_order_release 或更强内存顺序的原子存储是*释放操作*。互斥量上的 unlock() 操作也是释放操作。
    * 六种内存序：
        * memory_order_relaxed:对其他读取或写入没有施加同步或排序约束，仅保证此操作的原子性。配合std::atomic_thread_fence实现同步。
        * memory_order_consume (c++26中弃用一般也不推荐用)
        * memory_order_acquire：
            * 当前线程中，在它之后的所有读写操作都不能被重排序到这个 load 操作之前。
            * 同步效果：所有其他线程中，在释放（release）同一个原子变量之前的写操作，都会对当前线程可见。也就是说，如果你通过 acquire-load 读到了某个 release-store 的值，你就看到了那个线程在 release-store 之前做的所有事情。
        * memory_order_release：
            * 当前线程中，在它之前的所有读写操作都不能被重排序到这个 store 操作之后。
            * 同步效果：所有当前线程在 release-store 之前所做的写操作，对于那些通过 acquire-load 读取这个原子变量的线程是可见的。
        * memory_order_acq_rel：
            * 所有在它之后的操作不能重排到该操作前（就像 acquire）所有在它之前的操作不能重排到该操作后（就像 release）配合读-改-写 操作实现同步。
        * memory_order_seq_cst：
            * 提供了一个 全序关系（total order），所有 seq_cst 操作在*所有线程*中看起来是按同一个顺序执行的（全局顺序）。它既是 acquire 又是 release 操作。
    * 内存排序：
        * 宽松排序：标记为 memory_order_relaxed 的原子操作不是同步操作；它们不对并发内存访问施加顺序。它们仅保证原子性和修改顺序一致性。
        * 释放-获取排序：如果线程 A 中的原子存储标记为 memory_order_release，线程 B 中从同一变量进行的原子加载标记为 memory_order_acquire，并且线程 B 中的加载读取了线程 A 中的存储写入的值，则线程 A 中的存储同步于线程 B 中的加载。*同步仅在释放和获取同一原子变量的线程之间建立*。从线程 A 的角度来看，所有先序发生于原子存储的内存写入（包括非原子和宽松原子）都成为线程 B 中的可见的副作用。也就是说，一旦原子加载完成，就保证线程 B 可以看到线程 A 写入内存的所有内容。只有当 B 实际返回 A 存储的值或释放序列中稍后的值时，此承诺才成立。互斥锁（例如 std::mutex 或 原子自旋锁）是释放-获取同步的一个示例：当线程 A 释放锁并由线程 B 获取锁时，在线程 A 的上下文中，在临界区内发生的所有事情（在释放之前）都必须对线程 B 可见（在获取之后），线程 B 正在执行相同的临界区。*，release-acquire 同步可以有效地同步发生在 release 操作之前的所有内存操作（包括非原子操作）与发生在 acquire 操作之后的所有内存操作（也包括非原子操作）。这是 C++ 内存模型 std::memory_order_release 和 std::memory_order_acquire 的核心目的之一。效果同 fences*
        * 顺序一致性排序：标记为 memory_order_seq_cst 的原子操作不仅以与 release/acquire 排序相同的方式对内存进行排序（在一个线程中存储操作之前发生的所有内容都成为执行加载操作的线程中的可见副作用），而且还为*所有如此标记的原子操作*(而不仅仅是在同一原子变量间，所有的变量都是一致顺序的)建立单一的总体修改顺序(而 释放-获取 排序只有 获取线程 和 释放线程 两个线程中的特定 *一个原子变量上* 建立了统一的修改顺序(如果A->B A->C,B和C之间没有同步关系))。
* 5.3.4 release sequences and synchronizes-with
    * 可以在一个原子变量的*存储（store）*和另一个线程从该原子变量*加载（load）*之间获得一种同步关系，即使在存储和加载之间有一系列的*读改写*操作，只要所有的操作都被正确标记。在链中的任何原子*读改写*操作都可以*任意的内存排序*（甚至是 memory_order_relaxed）。
* 5.3.5 fences
    * 栅栏操作在不修改任何数据的情况下强制执行内存排序约束，通常与使用 memory_order_relaxed 排序约束的原子操作结合使用。
    * 栅栏的总体概念：如果一个获取操作看到了一个在释放屏障之后发生的存储的结果，那么这个屏障与该获取操作同步；如果一个在获取屏障之前发生的加载看到了一个释放操作的结果，那么这个释放操作与获取屏障同步。当然，您可以在两侧都有屏障，在这种情况下，如果一个在获取屏障之前发生的加载看到了一个在释放屏障之后发生的存储所写入的值，那么释放屏障与获取屏障同步。
    * 两项操作不被栅栏分隔，就不是有序的。只有当栅栏处于存储到x和存储到y之间时，它才会强加一个顺序。当然，栅栏的存在或缺失不会影响由于其他原子操作而存在的 happens-before 关系中的任何强制顺序。
    * 使用原子操作强制排序的真正好处是，它们可以对非原子操作强制排序，从而避免未定义的数据竞争行为。
    * atomic_thread_fence的三种同步
        * 根据 memory_order 的指示，建立非原子和宽松原子访问的内存同步顺序，而无需关联的原子操作。但请注意，如下所述，至少需要一个原子操作来建立同步。atomic_thread_fence 施加的同步约束比具有相同 std::memory_order 的原子存储操作更强。 虽然原子存储-释放操作阻止所有先前的读取和写入移动到存储-释放之后，但具有 std::memory_order_release 排序的 atomic_thread_fence 阻止所有先前的读取和写入移动到所有后续存储之后。栅栏-栅栏同步可用于向一系列宽松原子操作添加同步(具体例子和详细要求见cppreference atomic_thread_fence)
            * 1.栅栏-原子(获取)同步
                * 同时满足：1 存在原子存储 X（具有任何内存顺序），2 Y 读取由 X 写入的值（或者如果 X 是释放操作，则读取由以 X 开头的释放序列将写入的值），3 F 在线程 A 中先于 X 排序。
                * 在这种情况下，线程 A 中 先于 F 排序的所有非原子和宽松原子存储将 先于 线程 B 中在 Y 之后从相同位置进行的所有非原子和宽松原子加载。
            * 2.原子(释放)-栅栏同步：
                * 同时满足：1 存在原子读取 Y（具有任何内存顺序），2 Y 读取由 X 写入的值（或由以 X 开头的释放序列写入的值），3 Y 在线程 B 中先于 F 排序。
                * 在这种情况下，线程 A 中 先于 X 排序的所有非原子和宽松原子存储将 先于 线程 B 中在 F 之后从相同位置进行的所有非原子和宽松原子加载。
            * 3.栅栏-栅栏同步：
                * 同时满足：1 存在原子写入 X（具有任何内存顺序）在线程 A 中修改 原子对象M，2 FA 在线程 A 中先于 X 排序，3 线程 B 中存在原子读取 Y（具有任何内存顺序），4 Y 读取由 X 写入的值（或者如果 X 是释放操作，则读取由以 X 开头的释放序列将写入的值），5 Y 在线程 B 中先于 FB 排序。
                * 在这种情况下，线程 A 中 先于 FA 排序的所有非原子和宽松原子存储将 先于 线程 B 中在 FB 之后从相同位置进行的所有非原子和宽松原子加载。
        * 总结就是 存在一个原子变量M，一个线程中的释放操作X 和 另一个线程中的获取操作 Y 同步，存在 X 之前的栅栏FA 或 在 Y 之后的栅栏FB，那么FA之前的所有原子/非原子释放操作 可以和 Y后的所有原子/非原子加载操作同步，FB之后的所有原子/非原子释放操作 可以和 X 之前的所有原子/非原子加载操作同步，或者同时满足(范围这是FA之前的所有释放操作 和 FB之后的加载操作 )
* 5.3.6 ordering nonatomic operation with atomics
    * 通过使用原子操作来对非原子操作进行排序，此时 happens-before 关系中的 sequenced-before 关系就变得非常重要。如果一个非原子操作 sequenced-before 某个原子操作，而该原子操作又 happens-before 另一个线程中的某个操作，那么这个非原子操作也happens-before 另一个线程中的操作。
    * atomic_flog 可以实现自旋锁(spinlock),lock() 函数：使用 memory_order_acquire 顺序执行 flag.test_and_set() 循环；unlock() 使用 memory_order_release 顺序执行 flag.clear()。 当第一个线程调用lock()时，该标志最初是清除的，因此第一次调用 test_and_set() 将设置该标志并返回false，表示该线程现在拥有锁，并终止循环。此时调用lock（）的任何其他线程都将发现该标志已经设置，并将在 test_and_set() 循环中被阻塞。
