# 5基于锁的并发数据结构设计
* 选择用于编程问题的数据结构可能是整个解决方案的关键部分，并行编程问题也不例外。如果要从多个线程访问一个数据结构，要么它必须是完全不可变的，这样数据永远不会改变，也不需要同步，要么程序必须被设计成确保线程之间的变化是正确同步的。一种选项是使用一个单独的互斥锁和外部锁定来保护数据，采用我们在第三和第四章中研究的技术；另一种选择是将数据结构本身设计为支持并发访问。
* 在本章中，我们将从设计并发数据结构的一些一般准则开始。然后，我们将讨论锁和条件变量，并在继续讨论更复杂的数据结构之前，重新讨论这些基本数据结构的设计。
## 5.1并发数据结构设计 的含义
* 为并发设计数据结构意味着多个线程可以同时访问该数据结构，可以执行相同或不同的操作，并且每个线程将看到数据结构的一致视图。不会丢失或损坏任何数据，所有不变条件将得到维护，而且不会出现有问题的竞争条件。这种数据结构被称为线程安全。
* 一般来说，数据结构只对特定类型的并发访问安全。可能允许多个线程同时执行对数据结构的某种操作，而另一个操作则要求单个线程独占访问。或者，如果多个线程正在执行不同的操作，可能允许它们并发访问数据结构，而多个线程执行相同操作会导致问题。
* 然而，真正的并发性设计意味着更多：它意味着为访问数据结构的线程提供并发性的机会。互斥锁本质上是互斥的：一次只有一个线程可以获得互斥锁。互斥锁通过显式地阻止对它所保护的数据的真正并发访问来保护数据结构。这被称为序列化：线程轮流访问受互斥锁保护的数据。但在所有情况下，其思想都是一样的：受保护的区域越小，序列化的操作就越少，并发的可能性就越大。
* 在我们研究一些数据结构设计之前，让我们快速浏览一些简单的指导方针，了解在设计并发性时应该考虑什么。
* 5.1.1 并发数据结构设计的准则
    * 在为并发访问设计数据结构时，有两个方面需要考虑：确保访问是安全的，并确保能够真正的并发访问。第2章中介绍了如何使数据结构线程安全的基础：
        * 确保没有线程能够看到数据结构的不变性因另一个线程的操作而被破坏的状态。
        * 注意避免数据结构接口中固有的竞争条件，通过提供完整操作的函数而不是操作步骤。
        * 关注数据结构在异常出现时的行为，以确保不变性不会被破坏。
        * 最小化在使用数据结构时发生死锁的机会，通过限制锁的范围 和 尽量避免嵌套锁。
    * 在考虑这些细节之前，考虑对数据结构的用户施加哪些约束也很重要；如果一个线程通过特定的函数访问数据结构，那么从其他线程调用哪些函数是安全的？这个问题实际上非常重要。一般来说，构造函数和析构函数需要对数据结构进行独占访问，但用户需要自行确保在构造完成之前或在析构开始之后不进行访问。如果数据结构支持赋值、交换或复制构造，那么作为数据结构的设计者，您需要决定这些操作是否可以与其他操作安全地并发调用，或这些操作是否需要用户确保独占访问(即使大多数用于操作数据结构的函数可以在多个线程中并发调用而不会出现问题)。
    * 需要考虑的第二个方面是确保能够真正的并发访问。这里没有提供指导；相反，这里有一些问题，作为数据结构设计者进行自我提问。
        * 锁的作用范围是否可以限制，以允许操作的某些部分在锁外执行？
        * 数据结构的不同部分是否可以用不同的互斥锁保护？
        * 所有操作是否都需要相同级别的保护？
        * 是否可以通过简单改变数据结构来改善并发的机会，而不影响操作语义？
    * 所有这些问题都遵循一个思想：如何才能最小化必须发生的序列化量，并实现最大程度的真正并发性。数据结构允许仅读取数据结构的多个线程并发访问是很常见的，而可以修改数据结构的线程必须具有排他性访问。这可以通过使用boost:: shared_mutex这样的结构来支持。数据结构通常支持多线程并发执行不同操作，对执行的相同操作进行序列化(原文确实是serializing(但是应该理解为互斥))。
    * 最简单的线程安全数据结构通常使用互斥锁来保护数据。如在第2章中所见，尽管存在一些问题，但相对容易地确保了一次只有一个线程访问数据结构。
## 5.2基于锁的并发数据结构
* 基于锁的并发数据结构设计主要是*确保在访问数据时正确的锁定互斥锁，并确保锁的持有时间尽可能短*。当一个互斥锁保护一个数据结构，这就已经很困难了(您需要确保数据在互斥锁保护之外无法访问，并且接口中不存在竞争条件，正如您在第2章中看到的)。如果您使用单独的互斥锁来保护数据结构的不同部分，这些问题会变得更加复杂，并且如果对数据结构的操作需要锁定多个互斥锁，则很可能发生死锁。因此，您需要比设计单个互斥锁的数据结构时，更加小心地考虑具有多个互斥锁的数据结构的设计。
* 5.2.1 使用 locks 的线程安全的 stack
    * 查看每个指导方针，并了解它们如何在这里应用。
        * (不变量)首先，基本的线程安全性是通过对互斥锁保护每个成员函数来提供的。这确保了在任何时候只有一个线程实际上在访问数据，因此只要每个成员函数都维护不变量，就不会有线程看到不变量被破坏。
        * (接口中存在的竞态)其次，empty（）和pop（）函数之间可能存在竞争条件，但由于代码在持有pop（）中的锁时显式检查所包含的堆栈是否为空，因此这种竞争条件没有问题。通过将弹出的数据项直接作为pop（）调用的一部分返回，可以避免单独的top（）和pop（）成员函数（如std::stack<>中的成员函数）可能出现的竞争条件。
        * (异常安全)接下来，有一些潜在的异常来源。锁定互斥量可能会抛出异常。调用 data.push() B 可能会抛出异常。在 pop() 的第一个重载中，代码本身可能会抛出空栈异常。调用 std::make_shared 可能会因为无法为新对象分配内存而抛出异常，以及要返回的数据项的拷贝构造函数或移动构造函数在复制/移动到新分配的内存时可能会抛出异常。对 data.pop() 的调用 e 保证不会抛出异常，返回结果也是如此，因此该重载的 pop() 是异常安全的。pop（）的第二次重载与此类似。
        * (预防死锁)例子存在几个死锁的可能，因为在持有锁的情况下调用用户代码：复制构造函数或移动构造函数，以及对包含数据项的复制赋值或移动赋值运算符，还有可能是用户自定义的 new 运算符。如果这些函数在栈中调用了正在插入或移除的项的成员函数，或者要求任何类型的锁，而在调用栈成员函数时又持有了另一把锁，那么就有死锁的可能性。然而，合理的做法是要求栈的用户负责确保这一点；你不能合理地期待在不对其进行拷贝或分配内存的情况下将一个项目添加到栈中或从栈中移除它。
    * 此外，这个堆栈没有提供任何等待项被添加的手段，因此如果线程需要等待，必须定期调用 empty() 或者直接调用 pop() 并捕捉 empty_stack 异常。这使得这种堆栈实现成为一个不理想的选择，当用户需要等待一份数据时。将这种等待机制纳入数据结构本身的方法，通过在数据结构内部使用条件变量，因此让我们接下来看看这个内容。

* 5.2.3 使用细粒度 locks 和 condition_variables 的线程安全的 queue
    * 为了使用细粒度锁，需要看一下队列内部的组成结构，并且将一个互斥量与每个数据相关联。
    * 最简单的队列就是单链表。队列里包含一个头指针，其指向链表中的第一个元素，从队列中删除数据，其实就是将头指针指向下一个元素，并将之前头指针指向的值进行返回。队列里还有一个尾指针，该指针指向列表中的最后一个项。新节点通过将最后一项的下一个指针更改为指向新节点，然后更新尾指针以指向新项来添加。当列表为空时，头指针和尾指针均为 nullptr。
    * 在给定的实现中有两个数据项(head和tail)；即使，使用两个互斥量来保护头指针和尾指针，也会出现问题。最明显的问题就是push()可以同时修改头指针和尾指针，所以push()函数会同时获取两个互斥量。虽然会将两个互斥量都上锁，但这问题还不算太糟糕。糟糕的是push()和pop()都能访问next指针指向的节点：push()可更新tail->next，随后try_pop()读取read->next③。当队列中只有一个元素时，head==tail，所以head->next和tail->next是同一个对象，并且这个对象需要保护。不过，“在同一个对象在未被head和tail同时访问时，push()和try_pop()锁住的是同一个锁”就不对了。(带有虚拟节点解决这个问题)
