# 死锁

多个进程或线程互斥的持有并抢占导致的无限期地循环等待。

[操作系统关键知识点之死锁研究与总结 理论与实践回顾](https://blog.csdn.net/wj_rdk/article/details/148568284)

**死锁发生的四个条件，必须同时满足，死锁才会发生**

1. **互斥条件**：资源不能被共享，只能被一个线程占用。
2. **持有并等待**：线程持有一些资源，同时等待获取其他资源。
3. **不可剥夺**：已分配的资源不能被强制释放。
4. **循环等待**：存在一个线程的循环链，每个线程都在等待下一个线程持有的资源。
   
   

解决死锁的策略
-------

解决死锁的思路就是**破坏**上面提到的四个必要条件中的至少一个。

### 1. 破坏"循环等待"条件 - 锁排序

这是最常用且最有效的预防策略。**核心思想：** 为所有锁定义一个全局的获取顺序，所有线程都必须按照这个顺序来申请锁。

### 2. 破坏"请求与保持"条件 - 一次性申请所有资源

在开始执行任务前，一次性申请所需的所有锁，如果申请不到，就释放所有已获得的锁，稍后重试。

### 3. 破坏"不剥夺"条件 - 超时机制

使用 `pthread_mutex_trylock` 或带超时的锁，如果在一定时间内无法获得锁，就主动放弃已经持有的锁。

### 4. 死锁检测与恢复（高级策略）

对于一些系统（如数据库），它们会：

* **检测**：维护一个资源分配图，定期检查图中是否存在环。

* **恢复**：一旦检测到死锁，选择一个"牺牲者"进程，强制将其终止或回滚，从而释放其占用的所有资源。

## 总结

| 策略         | 原理        | 优点           | 缺点                |
| ---------- | --------- | ------------ | ----------------- |
| **锁排序**    | 破坏循环等待    | 简单有效，开销小     | 需要全局统一的顺序，复杂系统难维护 |
| **一次性申请**  | 破坏请求与保持   | 避免部分死锁       | 可能降低并发度，导致活锁      |
| **超时/尝试锁** | 破坏不剥夺条件   | 实践简单，能避免永久阻塞 | 编码复杂，可能造成饥饿       |
| **死锁检测**   | 允许发生，事后处理 | 灵活           | 实现复杂，开销大          |

**最实用建议：**

1. **首选锁排序**：在设计和代码审查阶段就规定好锁的获取顺序。

2. **保持锁的粒度尽可能小**：持有锁的时间越短，发生竞争和死锁的概率就越低。

3. **避免嵌套锁**：如果可能，尽量只持有一个锁。如果必须持有多个，务必小心。

4. **使用工具**：如 `helgrind(Valgrind中的一个工具)`、`tsan`（ThreadSanitizer）等工具可以帮助在运行时检测死锁和数据竞争。<https://zhuanlan.zhihu.com/p/446932523>

5. 在实际工程中，**“超时（time_mutex+try_lock） + 有序加锁 + 重试退避”** （同时实现）是一种常见且稳健的做法。

### **重试退避（Retry with Backoff）**

#### 目的：在竞争激烈时避免“忙等”或“活锁”，提高成功率

> 当因超时或冲突失败时，**不立即重试**，而是等待一段时间后再试，且等待时间可逐步增加（指数退避）。

#### 常见退避策略：

| 策略           | 描述                                      |
| ------------ | --------------------------------------- |
| **固定延迟**     | 每次失败后等待固定时间（如 100ms）                    |
| **随机退避**     | 在一定范围内随机延迟，避免“惊群效应”                     |
| **指数退避**     | 延迟时间按 2ⁿ 增长（如 10ms, 20ms, 40ms...），上限封顶 |
| **带抖动的指数退避** | 在指数基础上加随机扰动，更平滑（**推荐（指数+随机）**）          |

## 活锁 ：

活锁 vs 死锁
--------

为了更清晰地理解它们的区别，请看下表：

| 特征        | **死锁**        | **活锁**             |
| --------- | ------------- | ------------------ |
| **进程状态**  | 阻塞（Sleeping）  | 运行（Running）        |
| **资源占用**  | 持有资源并等待       | 可能不持有资源，或在不停地申请/释放 |
| **CPU使用** | 低（线程被阻塞）      | 高（线程在忙碌循环）         |
| **表现形式**  | 什么都做不了，程序"卡死" | 程序在"运行"，但无实际进展     |
| **根本原因**  | 循环等待资源        | 过度的“协作”或“重试”策略     |

产生活锁的常见场景
---------

1. **消息处理**：两个进程相互发送一条“重试”消息，对方收到后也回复一条“重试”消息，陷入循环。

2. **重试机制**：多个线程在失败后使用相同的退避策略，导致同时重试，同时失败，再次同时重试。

3. **资源分配**：多个线程都“礼貌地”让出资源给其他线程，又接着一起抢占失败，一直循环，结果谁都没用上资源。

解决和预防活锁的策略
----------

### 1. 引入随机性

在重试时加入随机延迟，打破同步循环。这是最有效的方法之一。

### 2. 使用退避算法

如指数退避，每次重试的等待时间指数级增加。

### 3. 引入协调者

让一个独立的协调者来决策，避免多个实体自作主张。

## 活锁检测

1. **基于性能指标的检测** 
    CPU使用率监控
   
   ```shell
   # 使用perf统计CPU周期
   perf stat -e cycles,instructions -p <pid> -- sleep 10
   # 如果CPU使用率持续很高但吞吐量为零，可能是活锁
   ```
   
    系统调用监控
   
   ```shell
   # 使用strace统计系统调用
   strace -c -p <pid>
    #使用perf监控系统调用频率
    perf record -e syscalls:sys_enter_* -p <pid> -- sleep 10
    perf report
   ```

2. 使用工具Valgrind + Helgrind
   
   ```shell
   # 使用Helgrind检测线程问题
   valgrind --tool=helgrind --history-level=full ./your_program
   # 检测数据竞争和锁顺序问题，这些可能导致活锁
   ```

### 总结

性能监控：高CPU + 零吞吐量 = 可能活锁

进度跟踪：在关键路径添加进度计数器

超时机制：为操作设置合理的超时时间

随机性引入：检测到可能的活锁时，随机改变行为

日志分析：记录重试次数和状态变化

## 银行家算法

银行家算法是操作系统中一个非常重要且经典的**死锁避免**算法。
核心思想：银行家算法的目标不是阻止死锁的发生后恢复，而是**在资源分配前进行判断，只允许不会导致系统进入不安全状态的分配请求**。

它基于这样一个前提：**进程在运行前应声明其所需的最大资源量**。
算法中的关键概念

假设系统有 `n` 个进程和 `m` 种资源。

1. **可用资源向量 Available**：一个长度为 `m` 的数组，表示每种资源当前可用的数量。  
   `Available[j] = k` 表示资源类型 `Rj` 有 `k` 个实例可用。

2. **最大需求矩阵 Max**：一个 `n x m` 的矩阵，定义了每个进程对每种资源的最大需求。  
   `Max[i][j] = k` 表示进程 `Pi` 最多需要 `k` 个资源类型 `Rj` 的实例。

3. **分配矩阵 Allocation**：一个 `n x m` 的矩阵，定义了每个进程当前已分配的每种资源的数量。  
   `Allocation[i][j] = k` 表示进程 `Pi` 当前已分配了 `k` 个资源类型 `Rj` 的实例。

4. **需求矩阵 Need**：一个 `n x m` 的矩阵，表示每个进程还需要的各类资源数量。  
   `Need[i][j] = k` 表示进程 `Pi` 还需要 `k` 个资源类型 `Rj` 的实例才能完成任务。  
   **重要公式：`Need[i][j] = Max[i][j] - Allocation[i][j]`

安全性算法（核心）

这是银行家算法的基石，用于判断系统当前是否处于**安全状态**。

**安全状态**：存在一个**安全序列** <P1, P2, ..., Pn>，使得按此序列为每个进程分配其所需资源，每个进程都能顺利完成。

**算法步骤**：

1. 设 `Work` 和 `Finish` 为长度分别为 `m` 和 `n` 的向量。
   
   * 初始化：`Work = Available` （当前可用资源）
   
   * `Finish[i] = false` （对于所有进程 i，初始化为未完成）

2. 寻找一个满足以下条件的进程 `i`：
   
   * `Finish[i] == false` （进程 i 尚未完成）
   
   * `Need[i] <= Work` （进程 i 所需的**所有**资源数量都小于等于当前可用资源）

3. 如果找到这样的进程 `i`：
   
   * 假设进程 `i` 完成任务并释放所有资源：  
     `Work = Work + Allocation[i]`
   
   * `Finish[i] = true`
   
   * 回到步骤 2

4. 如果所有进程的 `Finish[i] == true`，则系统处于**安全状态**；否则，系统处于**不安全状态**。

资源请求算法

当一个进程 `Pi` 发出一个资源请求向量 `Request_i` 时（`Request_i[j]` 表示进程 `Pi` 请求资源 `Rj` 的实例数），系统按以下步骤检查：

1. **检查请求是否超过声明**
   
   * 如果 `Request_i <= Need[i]`，继续步骤 2。
   
   * 否则，报错（请求超过了其声明的最大需求）。

2. **检查系统是否有足够资源**
   
   * 如果 `Request_i <= Available`，继续步骤 3。
   
   * 否则，进程 `Pi` 必须等待（因为资源不足）。

3. **系统尝试进行分配**（注意：这只是**试分配**）
   
   * `Available = Available - Request_i`
   
   * `Allocation[i] = Allocation[i] + Request_i`
   
   * `Need[i] = Need[i] - Request_i`

4. **调用安全性算法**，检查这次试分配后系统是否仍处于安全状态。
   
   * 如果安全，则正式完成资源分配。
   
   * 如果不安全，则本次分配被**拒绝**，系统回滚到步骤 3 之前的状态，进程 `Pi` 等待。

一个具体的数字例子

假设系统有3种资源：`A(10), B(5), C(7)`。  
有5个进程：P0, P1, P2, P3, P4。

在某一时刻，系统的状态如下：

| Process | Allocation (A,B,C) | Max (A,B,C) | Available (A,B,C) |
| ------- | ------------------ | ----------- | ----------------- |
|         |                    |             | (3, 3, 2)         |
| P0      | (0, 1, 0)          | (7, 5, 3)   |                   |
| P1      | (2, 0, 0)          | (3, 2, 2)   |                   |
| P2      | (3, 0, 2)          | (9, 0, 2)   |                   |
| P3      | (2, 1, 1)          | (2, 2, 2)   |                   |
| P4      | (0, 0, 2)          | (4, 3, 3)   |                   |

**第一步：计算 Need 矩阵 (Need = Max - Allocation)**

| Process | Need (A,B,C) |
| ------- | ------------ |
| P0      | (7, 4, 3)    |
| P1      | (1, 2, 2)    |
| P2      | (6, 0, 0)    |
| P3      | (0, 1, 1)    |
| P4      | (4, 3, 1)    |

**第二步：执行安全性算法**

1. `Work = (3, 3, 2)`

2. 寻找 `Need[i] <= Work` 的进程。
   
   * P1: `Need[1] = (1,2,2) <= (3,3,2)`? **是**。
   
   * 执行 P1：`Work = Work + Allocation[1] = (3,3,2) + (2,0,0) = (5,3,2)`, `Finish[1]=true`。

3. 新的 `Work = (5,3,2)`
   
   * P3: `Need[3] = (0,1,1) <= (5,3,2)`? **是**。
   
   * 执行 P3：`Work = (5,3,2) + (2,1,1) = (7,4,3)`, `Finish[3]=true`。

4. 新的 `Work = (7,4,3)`
   
   * P4: `Need[4] = (4,3,1) <= (7,4,3)`? **是**。
   
   * 执行 P4：`Work = (7,4,3) + (0,0,2) = (7,4,5)`, `Finish[4]=true`。

5. 新的 `Work = (7,4,5)`
   
   * P2: `Need[2] = (6,0,0) <= (7,4,5)`? **是**。
   
   * 执行 P2：`Work = (7,4,5) + (3,0,2) = (10,4,7)`, `Finish[2]=true`。

6. 新的 `Work = (10,4,7)`
   
   * P0: `Need[0] = (7,4,3) <= (10,4,7)`? **是**。
   
   * 执行 P0：`Work = (10,4,7) + (0,1,0) = (10,5,7)`, `Finish[0]=true`。

**结论**：因为所有 `Finish[i]` 都为 true，所以系统处于**安全状态**。其中一个**安全序列**是 <P1, P3, P4, P2, P0>。

优缺点

**优点**：

* 理论上可以有效避免死锁。

**缺点**：

* 需要提前知道进程的最大资源需求，这在现实中往往很难精确预估。

* 算法开销较大，需要维护多个矩阵并执行安全性检查。

* 进程数量和环境资源数量必须是固定的，缺乏灵活性。

因此，银行家算法更适用于一些**特定、封闭**的系统（如某些数据库管理系统），而在通用的操作系统中使用较少。
